# Тестовое задание Frontend разработчик на React.

Вам дан проект Дашборда со списком звонков, для операторов, в котором они могут просматривать детали звонка и менять их статусы.

```
Проект уже работает, но в нем намеренно допущены ошибки, которые вам предстоит выявить и исправить.
```

## Что нужно сделать:

- Найти и исправить ошибки в коде
- Правильно обработать ошибки от API
- Улучшить качество и читаемость кода

## На что стоит обратить внимание:

- Что может вызывать проблемы с производительностью
- Работа с React Query и обработка ошибок
- Работа с состоянием
- Типизация

## Чего делать не нужно:

- Полностью рефакторить код
- Тратить время на стилизацию

## Как отправить свое задание на проверку:

- Форкнуть репозиторий на GitHub
- Внести изменения в master ветке
- Описать в README какие баги и недочеты вы обнаружили, исправили и почему
- Отправить ссылку на репозиторий в GitHub

## Что было найдено и исправлено

1. Исправлена ошибка типизации в мутации смены статуса звонка.
   Причина: `react-query` ожидает, что `mutationFn` принимает один объект `variables`, а не несколько аргументов. Из-за этого вызов `updateStatus.mutate({ id, status })` конфликтовал.
   Что сделано: обновление статуса переведено на один типизированный объект `UpdateCallStatusInput`.

2. Убраны слабые места в типизации.
   Причина: в коде использовались нестрогие типы и обходы через `any`, что скрывало реальные ошибки и делало контракты между API, хуками и компонентами менее надёжными.
   Что сделано: добавлены и приведены в порядок типы для звонков и входных данных мутации, после чего проект прогнал  через `tsc` -- без ошибок.

3. Добавлена базовая обработка ошибок API.
   Причина: ошибка загрузки списка и ошибка смены статуса должны быть видимы пользователю
   Что сделано: для списка показывается `callsQuery.error.message` и кнопка `Retry`, для смены статуса показывается `updateStatus.error.message`.

4. Исправлена проблема с состоянием выбранного звонка.
   Причина: хранение целого объекта `selectedCall` в zustland приводило к риску рассинхронизации с данными из React Query после обновлений, refetch и socket-событий..
   Что сделано: в store теперь хранится только `selectedCallId`;

5. Исправлена проблема производительности списка.
   Причина: список из 1000 элементов рендерился через обычный `.map(...)`, из-за чего React создавал и обновлял слишком много DOM-узлов.
   Что сделано: список переведён на `react-window`, а `CallItem` обёрнут в `React.memo`, чтобы на экране рендерились только видимые элементы и уменьшилось число лишних перерисовок.

6. Исправлено небезопасное обновление данных из socket.
   Причина: частичное обновление звонка приводилось к полному объекту через небезопасный `as Call`, что могло замаскировать неполные данные.
   Что сделано: socket-обновление теперь аккуратно мержится с текущим объектом звонка.
